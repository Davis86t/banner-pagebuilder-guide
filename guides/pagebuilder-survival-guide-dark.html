üß† CORE MENTAL MODEL
What PageBuilder actually is
Big Picture Concept
  • Frontend: AngularJS 1.x + jQuery, auto-generated HTML & scopes.
  • Backend: Virtual Domains (PL/SQL) for data access.
  • Glue: Dataset objects (*DS) that handle load/save/paging.
  • Reality: A DSL on top of PL/SQL, not a modern React app.
Rule 0: Write code in a real editor, then paste into PageBuilder. Timeouts will eat unsaved work.
Where Things Live Map
  • Virtual Domains: DB-facing SQL / PL-SQL.
  • Resources: Glue between pages and virtual domains.
  • Grids / Data: Visual or headless data consumers.
  • CSS Manager: Shared styles; nothing magical.
VD = ‚Äúview + wrapped query‚Äù Grid / Data = dataset clients
When in Doubt Checklist
  1. Turn debug mode ON.
  2. Inspect the browser console for scope and datasets.
  3. Check roles / privileges before blaming SQL.
  4. Verify bind variables, case, and parameters.
Treat PageBuilder as ‚ÄúAngular + PL/SQL under a hood,‚Äù not a WYSIWYG form tool.
üóÑÔ∏è VIRTUAL DOMAINS (PL/SQL)
Data source layer
What They Are Definition
  • PL/SQL that defines the base SELECT for a resource.
  • PageBuilder wraps your SQL with paging and filtering logic.
  • Grids / Data components call these through resource objects.
Think: ‚Äúcurated query + PageBuilder-added paging wrapper.‚Äù
Bind Vars & Roles Gotchas
  • Bind variables must be lowercase: :sid, not :SID.
  • Uppercase bind vars ‚Üí null parameters ‚Üí empty / broken results.
  • Roles: missing role can block domain access before grids load.
Symptom: error as soon as page loads, grid never renders rows ‚Üí check roles and bind variable casing first.
Seeing the Real SQL Debug Trick
  1. Introduce a deliberate typo in the domain SQL.
  2. Trigger a grid/data load that uses it.
  3. Inspect the error response‚Äôs resource object.

The wrapped query there is what PageBuilder actually executed.

üí≤ IMPLICIT VARIABLES & SCOPE
How the $magic works
Compilation Pattern From $ to $scope

You write:

console.log($myTextbox);

PageBuilder compiles:

console.log($scope.myTextbox);
  • Each component defines properties on a specific Angular $scope.
  • Datasets (for grids/data) attach as *DS objects on that scope.
  • Implicit variables are simply syntactic sugar over $scope.
Inspecting the Scope Console Trick
  1. Add a button with an onClick script:
console.log(this);
  1. Click the button in the running page.
  2. In DevTools, this is the current $scope.

Look for:

  • this.myGridDS ¬∑ dataset object for a grid.
  • this.myTextbox ¬∑ value of a text input.
  • this.$externalScope ¬∑ alternate scope if $var appears undefined.
This is the fastest way to discover what PageBuilder actually generated for a page.
üìä DATASETS & LOADING
Grid/Data behavior
Dataset Naming & API *DS objects
  • Grid name: myGrid ‚Üí dataset: myGridDS.
  • PageBuilder suggests $myGrid.$load(), but that can break with scope changes.

Preferred pattern:

$myGridDS.load({ paging: true }, true);

Datasets expose:

  • data ‚Äì array of rows.
  • load() ‚Äì fetches rows.
  • dirty() ‚Äì unsaved changes.
  • inProgress ‚Äì load in flight.
‚ÄúEmpty Data‚Äù Race Condition Common Bug

If a user triggers .load() while a previous load is still running, data can end up empty.

Guard example:

if (!$myGridDS.inProgress) {
  $myGridDS.load({ paging: true }, true);
}
Also consider disabling buttons or controls while inProgress is true to avoid overlapping loads.
ü©∫ DEBUG MODE & QUICK FLOW
When things break
Debug Mode Must Use

Without debug, you usually see only a generic 500 error.

With debug enabled, you can see:

  • Compiled SQL (including PageBuilder‚Äôs wrapper).
  • Bind variable names and values.
  • PL/SQL exception messages.
  • Which resource / domain actually failed.
Do all development with debug ON; turn it OFF only for production runs.
Quick Triage Flow Checklist
  1. Enable debug mode.
  2. Load the page; capture errors.
  3. If domain errors appear immediately ‚Üí check roles and bind variables.
  4. If SQL errors show ‚Üí fix the PL/SQL or parameters.
  5. If no errors but no data ‚Üí inspect *DS.data and paging filters.
  6. Use a test button with console.log(this); to inspect scope.
This sequence covers 90% of ‚ÄúPageBuilder just broke‚Äù scenarios.
üöÄ QUICK START: TOUCHING AN EXISTING PAGE
Use this the first week
  1. Identify the page and its Object Name.
  2. Turn debug mode ON.
  3. Open Page Composer; copy existing scripts into your editor.
  4. Add a temporary button with console.log(this); to inspect scope.
  5. Find the relevant *DS (e.g., myGridDS) in the console.
  6. Trigger a manual load with $myGridDS.load({ paging: true }, true);.
  7. If data is wrong/empty:
    • Check Virtual Domain SQL + lowercase bind variables.
    • Verify roles / privileges.
    • Guard against overlapping loads using inProgress.
  8. Once stable, paste final code back into PageBuilder and remove debug helpers.
